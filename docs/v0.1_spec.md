# NexaFlow v0.1 版本规格

## 0. 当前实现状态

截至当前版本，v0.1 主链路已基本闭环：

- 已支持 mock、pcap_replay、Ubuntu live_pcap 三种采集模式。
- live_pcap 使用 Linux AF_PACKET 原始套接字，支持控制台选择网卡和简化 BPF 过滤。
- pcap_replay 支持经典 Ethernet `.pcap` 文件和回放倍率。
- 已支持 5 秒窗口聚合、Redis 实时 TopN、ClickHouse 历史写入。
- 已新增结构化 `flow_sessions_5s` 会话聚合表，默认每窗口保留 Top 500 五元组，并支持通过 `session_topn` 调整，用于会话追踪、公网访问和应用分析优先读取五元组字段。
- 已创建 1 分钟汇总表和物化视图：`link_traffic_1m`、`ip_traffic_1m`、`dimension_traffic_1m`。
- 已支持基于端口和协议的应用/服务识别维度，聚合 `service`、`service_category`、`service_risk` TopN。
- 已支持 VLAN ID、DSCP、ECN 解析和聚合维度，便于观察二层网段和 QoS 标记流量。
- Collector 已接入 Linux 网卡 RX/TX bytes、packets、dropped、errors 窗口增量，并通过 `capture_quality_5s` 独立记录接口采集质量。
- 已提供 `/healthz`、`/readyz`、`/metrics`。
- Collector 在线/离线状态基于最近 30 秒采集窗口推断，离线时会合成采集健康告警。
- Web 控制台已覆盖总览、实时监控、流量剖析、应用服务排行、TopN、IP/端口画像、服务拓扑、服务暴露、资产台账、风险线索、告警、检索、历史窗口和采集器配置。

仍未进入 v0.1 范围或仅预留的能力：

- PostgreSQL 仍作为部署依赖和低频配置层预留，当前核心配置主要使用运行时 JSON 和 ClickHouse 覆盖表。
- 采集接口统计已接入 Linux 网卡 RX/TX counters，但尚未细分为内核 socket 队列丢包、BPF 过滤前后数量和用户态处理积压。
- 更完整的 RBAC、审计、报表、sFlow/IPFIX、L7 指纹级应用识别属于后续 v0.5/v1.0 范围。

## 1. 版本定位

v0.1 是 NexaFlow 的端到端 PoC 版本，目标是验证“采集 -> 聚合 -> 存储 -> 查询 -> 展示 -> 告警”的主链路可行。

本版本不追求完整企业功能，也不承诺 10G 生产性能。优先让系统在单机 Ubuntu 环境中稳定运行，并能回答最核心的问题：

- 当前链路流量是多少？
- 哪些 IP 占用流量最多？
- 哪些目的端口、协议占用最多？
- 采集器是否正常，有没有丢包或处理积压？
- 是否出现基础阈值异常？

## 2. 明确目标

### 2.1 功能目标

- 支持单机部署。
- 支持模拟流量数据，用于本地开发和无网卡环境演示。
- 支持 pcap 文件回放，用于可重复测试。
- 支持 Ubuntu 上的基础网卡采集模式，优先 libpcap，后续替换或增强为 AF_PACKET。
- 支持 5 秒窗口聚合。
- 支持链路流量、IP TopN、端口 TopN、协议 TopN。
- 支持 ClickHouse 历史写入和查询。
- 支持 Redis 保存最近实时窗口和 TopN。
- 支持 PostgreSQL 保存采集源、告警规则等低频配置。
- 支持简单 Web Dashboard。
- 支持基础阈值告警。

### 2.2 验收目标

- 单机环境启动所有服务后，Dashboard 可正常打开。
- 模拟数据模式下，Dashboard 每 5 秒刷新流量曲线和 TopN。
- pcap 回放模式下，能从 pcap 文件生成 IP、端口、协议统计。
- Ubuntu 网卡采集模式下，能采集指定接口的基础 L3/L4 元数据。
- 最近 5 分钟 TopN 查询 P95 小于 1 秒。
- 24 小时内的分钟级趋势查询 P95 小于 3 秒。
- Collector 异常停止或采集丢包率超过阈值时产生告警事件。

## 3. 本版本不做

- 不做 DPDK、AF_XDP、SmartNIC。
- 不做 XDP/eBPF，最多预留接口。
- 不做 Inline 串接和阻断能力。
- 不做 TLS 解密。
- 不保存完整包正文。
- 不做复杂 DPI。
- 不做 Kubernetes Agent。
- 不做云 Flow Logs 适配。
- 不做多租户。
- 不做完整 RBAC，仅保留一个本地管理员用户或开发免登录模式。
- 不做复杂报表系统，仅预留导出接口。
- 不做高可用集群。

## 4. 推荐技术栈

### 4.1 后端

- Go：Collector、Aggregator、API Service。
- gopacket/libpcap：v0.1 采集和 pcap 回放。
- ClickHouse：历史聚合数据。
- Redis：实时窗口、TopN、告警流。
- PostgreSQL：配置、采集源、告警规则。

### 4.2 前端

- Vue 3。
- TypeScript。
- Vite。
- ECharts。
- Pinia。
- Vue Router。
- Naive UI 或 Ant Design Vue，二选一，建议先选 Naive UI，开发成本较低。

### 4.3 部署

- 本地 Mac：开发 API、前端、模拟数据、pcap 回放。
- Ubuntu Server：验证真实网卡采集。
- Docker Compose：启动 Redis、PostgreSQL、ClickHouse、API、Web。

```text

Collector
        |
        v

        |
        +--> Redis realtime
        |

        |
```

v0.1 暂不拆分 Collector 和 Aggregator 为独立进程，建议先做成一个 `collector` 服务，内部完成采集、解析、窗口聚合和写入。API 独立成 `api-server`。这样可以降低第一版进程间通信复杂度。

## 6. 采集模式

### 6.1 mock 模式

用途：

- Mac 本地开发。
- 前端联调。
- CI 测试。

行为：

- 每秒生成一批伪造的流量事件。
- 支持固定随机种子，方便测试结果稳定。
- 生成内网 IP、公网 IP、常见端口、TCP/UDP/ICMP 协议。

### 6.2 pcap replay 模式

用途：

- 离线测试。
- 回归验证。
- 演示已知流量样本。

行为：

- 从指定 pcap 文件读取包。
- 按原始时间戳或加速倍率回放。
- 只解析 Ethernet、VLAN、IPv4、IPv6、TCP、UDP、ICMP。

### 6.3 live pcap 模式

用途：

- Ubuntu 真实网卡验证。

行为：

- 使用 libpcap 打开指定接口。
- 支持 BPF filter。
- 只采集包头元数据。
- 统计采集包数、字节数、丢包数。

约束：

- macOS 可做开发验证，但真实链路验证以 Ubuntu 为准。
- 采集生产镜像口需要 root、`CAP_NET_RAW` 或容器特权权限。

## 7. 数据模型

### 7.1 PacketMeta

```go
type PacketMeta struct {
    Ts        int64
    SourceID  string
    Iface     string
    SrcIP     [16]byte
    DstIP     [16]byte
    SrcPort   uint16
    DstPort   uint16
    Proto     uint8
    VlanID    uint16
    DSCP      uint8
    Length    uint32
    TCPFlags  uint16
}
```

### 7.2 聚合窗口

默认窗口：5 秒。

每个窗口输出：

- `link_traffic_5s`
- `ip_traffic_5s`
- `dimension_traffic_5s`
- `alert_events`

v0.1 暂不长期保存完整五元组明细。当前保存可配置 TopN 结果，不保存全量 flow 表。

## 8. ClickHouse 表

v0.1 使用 3 类表：

- `link_traffic_5s`：链路级 bytes、packets、drops、utilization。
- `ip_traffic_5s`：按 IP 和方向聚合。
- `dimension_traffic_5s`：端口、协议等通用维度。

分钟级汇总先做物化视图：

- `link_traffic_1m`
- `ip_traffic_1m`
- `dimension_traffic_1m`

保留策略：

- 5 秒聚合：保留 7 天。
- 1 分钟聚合：保留 30 天。
- 告警事件：保留 180 天。

```text
rt:link:{source_id}:{window_ts}
```

## 11. 前端页面

### 11.1 Dashboard

内容：

- 当前入/出方向 bps。
- 当前 pps。
- 活跃 IP 数。
- 当前告警数。
- 最近 5 分钟流量趋势。
- Top 源 IP。
- Top 目的 IP。
- Top 目的端口。
- 协议占比。
- Collector 健康状态。

### 11.2 Realtime

内容：

- 秒级或 5 秒级吞吐曲线。
- pps 曲线。
- 采集包数、字节数、丢包数。
- Collector heartbeat。
- 最近窗口 Top Talkers。

### 11.3 TopN

过滤：

- 时间范围。
- 采集源。
- 维度：源 IP、目的 IP、目的端口、协议。
- 指标：bytes、packets。
- Top N：10、20、50、100。

### 11.4 Alerts

内容：

- 告警列表。
- 严重程度。
- 状态。
- 主体。
- 首次出现、最近出现。
- 确认和关闭操作。

## 12. 告警规则

v0.1 只做阈值规则：

- 链路利用率超过 80% 持续 5 分钟。
- 单 IP 在 5 分钟内流量超过配置阈值。
- 采集丢包率超过配置阈值。
- Collector heartbeat 超过 30 秒未更新。

暂不做：

- 基线告警。
- 扫描检测。
- DDoS 行为模型。
- 可疑外联评分。

```yaml
  enabled: true
  link_utilization_threshold: 0.8
  drop_rate_threshold: 0.01
```
